## 0. 你的任務一句話
幫一位老師把「3對3女生籃球」的多場**手寫攻守紀錄表掃描檔**，辨識後彙整成每位同學的累積統計總表。核心痛點是**資料輸入與校對**，不是漂亮畫面。要做到：**圖片辨識 → 草稿入庫 → 老師逐張校對 → 按「確認」即自動寫入本地 DB → 匯出統計總表**。

## 1. 為什麼從 Cowork 移植到你(Claude Code)
Cowork 版用純 HTML 檔，瀏覽器沙箱**不能自動寫入本地檔案**，所以「按確認即自動入庫」做不到（只能存 localStorage 再手動匯出）。使用者要的是**確認即自動同步到本地 DB**。這需要一個會跑的本地後端，正是你擅長的：
- 建一個小型本地服務（建議 **Python FastAPI 或 Flask + SQLite**，或 Node + better-sqlite3）。
- 前端沿用已做好的「閃卡校對」介面，把「確認」改成 `POST` 到後端 → 寫 SQLite。
- 提供一鍵啟動腳本（例如 `start.command` / `run.py`），讓非工程師也能用。

> 註：SQLite 不是硬需求，JSON 檔也夠用（單人、約 33 場）。關鍵是「確認即持久化」與「彙整查詢」。但使用者有提到 SQLite，以下給了 schema。

> **關鍵動機（使用者明確點出）**：開發過程會**滾動式更新前端介面**（例如每辨識 10 張就重發一版 HTML）。純前端把資料存在 localStorage 時，換一個新的 HTML 檔（尤其 `file://`）常讀不到舊資料 → 資料斷掉。**因此務必讓資料住在 DB、與前端完全脫鉤**：前端是可隨時丟棄/改版的，已校對資料永遠在 SQLite 裡，一場都不能掉。

## 5. 統計總表目標欄位（最終交付，對應老師原本的 Excel）
`序號, 姓名, 完賽場數, 勝場次, 負場次, 總得分, 助攻, 抄截, 防守籃板, 進攻籃板, 犯規, 擔任主裁判, 擔任副裁判, 擔任紀錄`
- 每位球員：累加各場數據；勝/負由所屬隊伍當場總分決定；裁判三欄各自計次。

## 6. 正字(計分)辨識的關鍵規則 ⚠️
紀錄表「得分」欄是**正字計數**(一個「正」=5 筆)，「總得分」欄是寫好的數字。兩者要互相對帳：
1. 設 `tally` = 得分欄正字筆數，`total` = 總得分數字。
2. `tally == total` → 正字直接代表分數，採用 `total`。
3. `tally * 2 == total` → **正字代表「投球次數」，得分 = 次數 × 2**（兩分球）。
4. 兩者都對不上 → **很可能有三分球或誤判**，標紅(`conflict`)請老師確認。
5. 沒有正字、只有總得分 → 直接採用 `total`，但若無從對帳則標為不確定。
6. **有人會投三分球**，所以 ×2 不一定對；任何不確定都要在 UI 標示（橘色「?」=可能三分/不確定）。
- 彙整用的「得分」以**總得分(total)為準**；正字僅作交叉檢查。

## 7. 資料模型（建議 SQLite schema）
```sql
CREATE TABLE players (name TEXT PRIMARY KEY);              -- 名單
CREATE TABLE games (
  id INTEGER PRIMARY KEY,
  no INTEGER, venue TEXT, date TEXT,
  ref_main TEXT, ref_asst TEXT, ref_scorer TEXT,
  page INTEGER, idx_on_page INTEGER,                       -- 來源頁/第幾份
  img_path TEXT,                                           -- 切圖路徑
  confirmed INTEGER DEFAULT 0                              -- 老師是否已確認
);
CREATE TABLE stat_lines (
  id INTEGER PRIMARY KEY,
  game_id INTEGER REFERENCES games(id),
  team TEXT,            -- 'A' / 'B'
  col INTEGER,          -- 欄位序 1..6
  name TEXT,
  name_uncertain INTEGER DEFAULT 0,
  tally INTEGER,        -- 正字筆數(可空)
  total INTEGER,        -- 總得分數字(可空)
  points INTEGER,       -- 對帳後採用的得分
  ast INTEGER, stl INTEGER, oreb INTEGER, dreb INTEGER, foul INTEGER, tfoul INTEGER,
  conflict INTEGER DEFAULT 0,   -- 正字↔總得分對不上
  maybe3 INTEGER DEFAULT 0      -- 可能三分/不確定
);
```
彙整查詢：以 `confirmed=1` 的 games 為準，依 stat_lines 累加；勝負比較同場 A/B 的 `SUM(points)`；裁判計次掃 games 三欄。

## 9. 建議的工作流程（滾動式修正）
1. 切圖 → 你(Claude Code 可讀圖)逐張辨識 → 寫入 DB(`confirmed=0`)，套用第 6 節正字規則並標旗標。
